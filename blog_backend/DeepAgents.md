# 三、必须补的一块：为什么用 Agent

这个是面试高频题。你 README 里现在没体现。

你需要专门加一节：

## 为什么采用 Agent Workflow

可以写成：

> 传统规则爬虫适合页面结构固定、字段规则明确的场景，但本项目面临以下问题：
>
> * 信息来源多样，网页结构不统一
> * 同一项目在不同来源中的表述存在差异
> * 目标字段并非固定模板可直接提取
> * 环评报告检索依赖前一步提取出的项目名称、建设单位等上下文信息，属于多跳任务
>
> 因此，本项目采用基于 LangGraph 的多阶段 Agent Workflow，将任务拆分为相关性判断、项目抽取、动态搜索、报告下载与解析等多个节点，通过 LLM 驱动决策与工具调用，提高系统在复杂网页环境下的适应能力。

这段一写，面试官就会觉得你不是“为了炫技而上 Agent”。

# 四、你的工作流还要再“工程化”一点

你现在的 5 个步骤已经有了，但还不够“系统设计感”。

建议你把每一步写成：

* 输入是什么
* 输出是什么
* 失败怎么办
* 为什么这样设计

比如这样：

## Agent Workflow 设计

### 1. crawl

* 输入：新闻详情页 URL
* 输出：正文 HTML / 纯文本内容
* 作用：统一不同来源网页的原始内容，为后续分析提供标准化输入
* 风险点：网页结构差异、反爬限制、正文提取不完整

### 2. filter

* 输入：网页正文
* 输出：是否与声屏障 / 环评目标相关
* 作用：在进入高成本分析流程前先过滤无关内容，降低推理成本
* 风险点：LLM 误判导致召回不足或误召回

### 3. extract

* 输入：相关网页文本
* 输出：项目名称、投资总额、工程量、建设单位、搜索 Query
* 作用：将自然语言公告转换为结构化项目信息，并生成下一步检索线索

### 4. search

* 输入：结构化项目字段 + 搜索 Prompt
* 输出：候选环评报告链接
* 作用：通过搜索工具进行外部知识补全，实现多跳检索

### 5. download

* 输入：候选 PDF 链接
* 输出：本地 PDF 文件 + 解析结果
* 作用：从原始环评报告中抽取更完整、更可信的项目信息

这个写法会让人觉得你是在设计系统，不是在列流程。

这句话很加分。

---

## 2. 失败重试 / fallback

你现在最缺这个。

至少补这几种：

* 搜索不到 PDF 时，是否切换 query 重试
* 下载失败时，是否换候选链接
* PDF 解析失败时，是否回退到网页信息
* 提取字段不全时，是否二次搜索补充

你可以写：

> 为提升流程鲁棒性，系统设计了多级 fallback 机制：当首次搜索未命中目标 PDF 时，基于建设单位、项目名称等字段重写 Query 再次检索；当 PDF 下载或解析失败时，回退至网页公告中的结构化结果，避免整条链路中断。

这就是“企业级味道”。

# 六、最强补分项：做一个小型评估

这个真的很重要。

如果你说“感觉还行”，基本就没了。

你至少做一个简单评估。

## 你可以评估 4 个指标

### 1. Filter 准确率

从 30~50 条新闻里人工标注相关/不相关，统计准确率

### 2. Extract 字段完整率

统计项目名称、建设单位、投资金额等字段提取成功率

### 3. PDF 命中率

从相关样本中统计最终找到目标环评报告的比例

### 4. 端到端成功率

一条新闻从抓取到下载解析成功的比例

哪怕数据不大，也很有说服力。

你可以在 README 里加个表：

| 指标               | 说明                         | 当前结果 |
| ------------------ | ---------------------------- | -------- |
| 相关性过滤准确率   | LLM 判断新闻是否相关         | 85%      |
| 项目字段提取完整率 | 关键字段成功抽取比例         | 78%      |
| 环评报告检索命中率 | 成功找到目标 PDF 的比例      | 62%      |
| 端到端成功率       | 从抓取到解析完成的整体成功率 | 55%      |

哪怕这些数字还不高，也没关系。
重点是你有评估意识。

# 七、你还缺“技术难点”这一节

这节非常适合简历和面试。

建议你加 3~4 个难点，每个都写“问题 + 解决方法”。

例如：

## 技术难点与解决方案

### 1. 网页来源异构，结构不统一

* 问题：不同政府网站和新闻站点 HTML 结构差异大，难以用统一规则抽取
* 解决：crawl 节点仅负责正文提取与清洗，将结构理解交由后续 LLM 节点完成

### 2. 公告文本表达不规范，字段难以规则提取

* 问题：同一字段在不同公告中表述方式差异大
* 解决：通过 LLM 提取结构化项目字段，并结合 prompt 约束输出 schema

### 3. 环评报告检索属于多跳任务

* 问题：原始网页通常不直接提供 PDF 链接，需要基于项目线索进行外部搜索
* 解决：extract 节点动态生成搜索 query，再调用搜索工具进行候选链接检索

### 4. 长链路任务容易在中间节点失败

* 问题：搜索失败、下载失败、解析失败会影响整条链路
* 解决：增加候选链接回退、query 重写和失败状态记录机制

这类内容会比“用了 LangGraph / MiniMax / DashScope”更能体现能力。
