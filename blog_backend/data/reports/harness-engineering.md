# Harness Engineering（驾驭工程）—— 让 AI 稳定可靠地工作

## 一、引言

2026年初，HashiCorp 联合创始人、Terraform 创造者 Mitchell Hashimoto 首次正式定义了 **Harness Engineering** 这个概念。这并非一个凭空冒出来的新名词，而是 AI 使用方式演进到第三阶段的必然产物——从 Prompt Engineering（提示词工程）到 Context Engineering（上下文工程），再到 Harness Engineering（驾驭工程）。

它的核心命题是：当 AI Agent 拥有强大的代码生成能力后，如何确保其输出的**可靠性、一致性和长期可维护性**。

## 二、Harness Engineering 的起源

### 2.1 一个关键事件

OpenAI 在 GitHub 上启动了一个实验性项目：3 人工程师团队，5 个月时间，借助 Codex Agent 构建了超过 **100 万行代码**的产品，**零行人工手写代码**，人均日合并 3.5 个 PR，效率约为传统模式的 **10 倍**。

但真正引爆行业思考的，是负责这个项目的工程师 Ryan Lopopolo 的一句话：

> **"Agents aren't hard; the Harness is hard."**
> （Agent 不难，难的是 Harness。）

### 2.2 为什么需要 Harness？

AI（大语言模型）本质上是一个超级模式匹配器。它写代码、翻译、整理信息的速度和质量令人惊叹，但它**不可靠**——会编造事实、犯逻辑错误，且不"理解"自己在说什么。

在"一问一答"的聊天场景下，问题不大——人全程盯着，犯错了马上能发现。但当 AI 以 **Agent** 的形式自主执行任务——读代码、写代码、运行测试、修 bug、提交修改，整个过程可能持续数小时，中间无人介入——一个会犯错的东西自主运行数小时，风险急剧放大。

行业数据显示，大约 **80%-90% 的 AI Agent 项目没能进入生产环境**。原因不是 AI 不够聪明，而是没有人搭好它"干活的环境"[1][2]。

## 三、从 Prompt Engineering 到 Harness Engineering 的演进

### 3.1 第一阶段：Prompt Engineering（2022-2024）

2022 年底 ChatGPT 横空出世后，所有人都在研究如何精心设计提示词，让 AI 的输出更符合需求。但它只解决了"单次交互"的问题。要让 AI 自主完成一个 20 步的任务，不能给每一步都设计提示词。

### 3.2 第二阶段：Context Engineering（2025）

Shopify CEO Tobi Lütke 和 AI 专家 Andrej Karpathy 推广了这一概念——核心是"把恰好正确的信息，在恰好正确的时刻，以恰好正确的格式，填进 AI 的上下文窗口"[2]。通过 RAG（检索增强生成）管理对话历史和背景信息。

但即使给了 AI 正确的资料，它**依然可能跑偏**——资料告诉了它"应该做什么"，但没有告诉它"**绝对不能做什么**"。

### 3.3 第三阶段：Harness Engineering（2026）

核心理念来自 Mitchell Hashimoto 的定义：

> **"每次发现 Agent 犯了一个错误，你就花时间设计一个方案，让 Agent 再也不会犯同样的错误。"**

这不是替代前两个阶段，而是**嵌套包含关系**：Harness Engineering 包含 Context Engineering，Context Engineering 包含 Prompt Engineering。[1]

## 四、Harness Engineering 的四大支柱

业界普遍将 Harness Engineering 归纳为四大核心支柱[1][2]：

### 4.1 支柱一：约束（Constrain）——给 AI 画红线

AI Agent 能做的事情越多，它能闯的祸也越大。约束的作用是在 AI 开始工作前画好红线：

- **权限控制**：AI 只能读写特定的文件和目录，不能碰核心配置
- **操作限制**：AI 不能执行删除操作，不能直接操作生产环境数据库
- **架构边界**：AI 生成的代码必须遵守项目的分层结构。OpenAI Codex 团队规定代码依赖只能按固定方向流动：类型定义 → 配置 → 数据层 → 业务逻辑 → 运行时 → 界面，**不能反向依赖**

这些规则硬编码在系统中强制执行，AI 试图违反时操作直接被拒绝。[1]

### 4.2 支柱二：告知（Inform）——给 AI 递正确的资料

最典型的实践是 **AGENTS.md** 文件——项目根目录下专门给 AI Agent 读的"项目须知"。AI 开始工作前会先读取这个文件。

AGENTS.md 的设计原则是**精简**——约 100 行，每行都是指向深度文档的指针。过长会挤占任务 Token 导致性能下降。[2]

```
AGENTS.md 示例（简化）：
- 项目概述：TypeScript Web 应用，包管理器用 pnpm
- 常用命令：pnpm install / pnpm test / pnpm build
- 代码规范：TypeScript 严格模式，单引号，不加分号
- 绝对禁止：不允许提交 API 密钥，不允许使用 var
```

分成两层：根目录放基础规则，各子目录放特定领域规则，AI 在某目录工作时向上查找叠加。[1]

### 4.3 支柱三：验证（Verify）——自动检查 AI 的产出

全部自动化检查体系：

- **Linter**：自动扫描代码是否符合格式规范、命名约定
- **类型检查**：确认数据类型是否匹配
- **单元测试**：针对每个功能模块运行预设用例
- **构建验证**：整个项目编译一遍，确认没有语法错误
- **结构测试**（OpenAI 创新）：验证代码架构是否合规[1]

### 4.4 支柱四：纠正（Correct）——错了就自动修

Harness Engineering 的关键创新在于把错误信息自动反馈给 AI，让它自行修复，构成一个闭环：

```
AI 生成代码 → 自动检查 → 发现错误 → 错误信息传回 AI → AI 修复 → 再次检查 → 通过 → 合并
```

OpenAI 做了一个精妙设计：Linter 报告错误时，不只说"这行有错"，而是在错误信息里**直接嵌入修复指引**，这段文字被自动注入 AI 上下文——**不需要人类介入**。[1]

更关键的是纠正 Harness 本身：AI 犯了新类型的错误，就更新 AGENTS.md、增加 Linter 规则、写新的结构测试——**让这个错误在系统层面不可能再次发生**。

## 五、核心公式：Agent = Model + Harness

业界总结出了一个核心公式[2]：

```
Agent = Model + Harness
```

- **Model**（模型）：大语言模型，提供智能，非确定性，无状态
- **Harness**（驾驭具）：约束机制、反馈回路、状态管理、验证机制、工具调度——让智能有用

Harrison Chase（LangChain CEO）的比喻非常形象：**"模型是 CPU，Harness 是操作系统——CPU 再强，OS 拉胯也白搭。"**[2]

## 六、关键实践与发现

### 6.1 OpenAI 的反常识发现

团队从 3 人扩展到 7 人时，**人均效率不降反升**。这在传统软件开发中几乎不可能（Brooks 法则）。原因在于新加入的工程师不需要"学代码"，而是学"怎么设计 Harness"——而 Harness 的规则是显式的、可共享的、可复用的。[1]

这 7 名工程师的日常工作已经从"写代码"转变为：
- 设计架构约束
- 编写和维护 Linter 规则
- 更新 AGENTS.md
- 审查 AI 提交的 Pull Request
- 处理 AI 搞不定的边界情况

**他们从"代码的作者"变成了"AI 工作环境的设计师"。**

### 6.2 LangChain 的验证

LangChain 2026 年 3 月的报告《The Anatomy of an Agent Harness》表明：仅仅是给**同一个大语言模型**换上一套更精巧的 Harness 架构，在 Terminal Bench 2.0 上的表现直接起飞——**不换模型，只换环境，性能暴涨**。[2]

### 6.3 熵管理（Entropy Management）

AI 生成的代码库随时间推移会自然累积"熵"——文档与代码脱节、命名规范漂移、死代码堆积。OpenAI 的解决方案是建立一组**后台垃圾回收 Agent**周期性运行：

- **文档一致性 Agent**：验证文档与当前代码是否匹配
- **约束违规扫描器**：发现绕过早期检查的约束违反
- **模式执行 Agent**：识别并修复偏离既定模式的代码
- **依赖审计器**：追踪并解决循环或不必要的依赖[2]

## 七、适用场景与局限

### 适合的场景

- **编程领域**：代码有明确对错标准（能编译、能通过测试、不破坏架构），天然适合自动化验证
- **数据处理**：ETL 流程、数据清洗等有确定性输出的任务
- **测试自动化**：用例设计、回归测试、覆盖率检测

### 不适合的场景

- **创意性工作**：文案、设计等"什么算好"本身就主观
- **战略决策**：涉及模糊判断、商业直觉的领域
- **安全敏感场景**：Harness 本身的正确性谁来保证？

### 关键风险

- Harness 设计不当可能比没有 Harness 更糟——错误的约束会让 AI 在"合法"的框架内做出更隐蔽的错误
- Harness 的维护成本：规则库需要持续更新，否则会"腐化"
- 随着 AI 模型能力提升，某些 Harness 规则可能变得冗余[1]

## 八、总结

Harness Engineering 本质上不是什么全新的东西——它就是软件工程里"持续改进"的老思想，只不过管理对象从"人类程序员"变成了"AI Agent"[1]。

- **Prompt Engineering** 教你怎么跟 AI 说话
- **Context Engineering** 教你给 AI 看什么资料
- **Harness Engineering** 教你搭什么样的环境，让 AI 在里面安全、可靠、持续地干活

AI 时代，最值钱的能力可能不是"会用 AI"——而是**会给 AI 搭脚手架**。
